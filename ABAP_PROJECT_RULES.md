# ABAP Project Rules

## 目标
本项目使用 AI 辅助进行 ABAP 开发、重构、排障和测试生成。
AI 的目标是提升效率，但不能破坏业务语义、SAP 标准约束和系统安全。

## 基本原则
1. 先理解，再修改。
2. 先小范围变更，再扩大范围。
3. 先给方案，再写代码。
4. 先生成测试，再做高风险重构。
5. 任何涉及传输、激活、删除、权限、生产相关操作，必须先明确确认。

## 代码规范
1. 优先使用清晰命名，不使用难懂缩写，除非是 SAP 领域通用缩写。
2. 方法应保持单一职责。
3. 避免超长方法和过深嵌套。
4. 优先使用可测试、可分解的结构。
5. 新代码必须尽量保持与现有项目风格一致。
6. 不要随意重命名已有公共接口、字段、表结构、类名，除非明确要求。

## ABAP 约束
1. 必须考虑授权检查。
2. 必须考虑事务一致性。
3. 必须考虑异常处理。
4. 必须考虑锁、提交、回滚、更新任务等影响。
5. 必须考虑性能，尤其是大数据量 SELECT、内表循环、数据库往返。
6. 不得引入会破坏 Clean Core 的不必要改动。
7. 如果可以通过扩展点、BAdI、RAP、CDS、接口封装完成，不优先改标准对象内部逻辑。

## 修改策略
1. 优先最小变更。
2. 如果任务较大，先拆成多个步骤。
3. 每次只修改一个清晰边界内的对象。
4. 修改完成后，必须说明修改内容、影响范围、风险点和验证方式。
5. 如果存在多个方案，先给推荐方案，再给备选方案。

## 测试要求
1. 新增或修改关键逻辑时，尽量生成或更新 ABAP Unit。
2. 对异常路径、边界值、空值、权限失败、无数据场景要有覆盖。
3. 若无法写自动化测试，必须给出手工验证步骤。
4. 测试应尽量避免依赖真实外部系统。

## 输出要求
每次回答尽量按以下结构：
1. 结论
2. 变更方案
3. 代码或伪代码
4. 测试建议
5. 风险与回滚点
6. 需要确认的问题

## 高风险操作要求
以下操作必须先获得明确确认：
- 改动生产相关逻辑
- 删除对象
- 大范围重构
- 修改授权逻辑
- 修改传输策略
- 修改接口契约
- 修改数据模型
- 修改影响批量作业/定时任务的逻辑

## AI 行为要求
1. 不要编造 SAP 对象、表名、字段名、接口名。
2. 如果信息不足，先提出澄清问题。
3. 如果不确定，请明确标注不确定点。
4. 如果代码中有潜在问题，不要忽略，要主动指出。
5. 在不确定时，优先保守方案。
